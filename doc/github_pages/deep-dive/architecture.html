Lifecycle Manager

The LifecycleManager is the central orchestration mechanism. It manages ordered bring-up and tear-down of LifecycleComponent instances across numbered run levels (1–9).

Functional areas are encapsulated in "XyzSystem" classes. Each system owns its subsystem objects and wires them together in init() / run().

Run Level Ordering (typical) Level 1 RuntimeSystem + SafetySystem ← core infra first Level 2 BspSystem + CanSystem ← hardware drivers Level 4 TransportSystem ← transport layer Level 5 DoCanSystem + EthernetSystem + Storage Level 6 DoIpSystem Level 7 UdsSystem ← diagnostics Level 8 SysAdminSystem Level 9 DemoSystem ← user application Shutdown → Level 0 reached → flush logger → softwareSystemReset()

Registration Pattern

The generic app::startApp() registers platform-independent systems. Each platform implements platformLifecycleAdd(lifecycleManager, level) to inject its own BSP/CAN systems at the right run level. This is the key extensibility mechanism — no #ifdefs in the generic layer.

// Generic: app.cpp
void startApp() {
    lifecycleManager.addComponent("can",   canSystem,   2);
    lifecycleManager.addComponent("uds",   udsSystem,   7);
    lifecycleManager.addComponent("demo",  demoSystem,  9);
    platformLifecycleAdd(lifecycleManager, ...);  // ← platform callout
}

// Platform: nucleo_g474re/main.cpp
void platformLifecycleAdd(LifecycleManager& lm, ...) {
    lm.addComponent("bsp",    bspSystem,    2);
    lm.addComponent("safety", safetySystem, 1);
}

Key Design Decisions

No Vendor HAL

All BSP drivers use direct register-level access via CMSIS headers. No STM32 HAL, no NXP SDK. Direct RCC->AHB2ENR |= ... style manipulation. Keeps binary small, deterministic, and auditable.

Static Allocation

No heap usage. etl::typed_storage<> for lazy-constructed objects. LifecycleManager templated with compile-time max counts (MaxNumComponents=16, MaxNumLevels=9).

RTOS Abstraction

The async module abstracts over FreeRTOS and ThreadX. Maps "contexts" to RTOS tasks. AsyncBinding provides the concrete adapter per platform.

Compile-time Feature Toggles

Features are opt-in via PLATFORM_SUPPORT_CAN, PLATFORM_SUPPORT_ETHERNET, PLATFORM_SUPPORT_UDS, etc. Platforms only link what they support.

Interface-based Decoupling

Platform systems implement interfaces (ICanSystem, IEthernetDriverSystem). The generic app layer accesses them via extern getters: systems::getCanSystem().

Safety Partitioning

Safety split into 4 layers: generic (libs/safety/), platform-specific, project-specific, project+platform. Each layer adds its own safety checks.
